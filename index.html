By Ajay Singh Open original LinkedIn post ↗

Isolating Agentic Code: A Guide to Infrastructure for AI Agents | Ajay Singh posted on the topic | LinkedIn

Post summary

Everyone is talking about AI agents writing code. Far fewer are talking about the infrastructure required when they actually execute it.

If you give an LLM an autonomous execution environment, standard containerization is no longer enough. Agentic code is non-deterministic and inherently untrusted.

If you are building agentic workflows, here is a breakdown of the isolation tooling landscape—and exactly when to use (or avoid) each approach:

1. MicroVMs (Hardware-Level Isolation) The Tools: E2B, AWS Firecracker. MicroVMs boot their own kernel in ~150ms. If an agent executes a kernel exploit, it only crashes its own isolated instance. ✅ When to use: Ephemeral, high-throughput agent execution. Perfect for multi-step task loops where you need fast, highly isolated environments that destroy themselves immediately after the run. ❌ When NOT to use: If your agent requires massive, persistent workspaces across weeks, or if you need heavy GPU-accelerated workloads (unless you are prepared to manage complex self-hosted PCIe passthrough).

2. Userspace Kernels & Persistent Workspaces The Tools: Daytona, Northflank, Agent Sandbox (gVisor). These tools intercept system calls before they hit the host kernel, or manage persistent, stateful pods that can pause and resume. ✅ When to use: "Dev-in-a-box" agents. Use this if your agent needs to maintain state across a massive repository, install dependencies over multiple turns, and retain its environment over days. ❌ When NOT to use: When your primary risk is bare-metal performance. Abstraction layers like gVisor add overhead and can limit raw hardware capabilities.

3. Standard Containers The Tools: Default Docker. ✅ When to use: Local development, testing your orchestration framework, or running fully trusted, static scripts. ❌ When NOT to use: Production. Containers share the host kernel. Giving an autonomous agent root inside a standard container is a direct path to a sandbox escape.

The Golden Rule: Layer 1 isolation (the sandbox) is just the start. You still need Layer 2 guardrails: default-deny network egress, short-lived ephemeral credentials, and hard compute timeouts.
